# /dcad-design — DynamoDB 전환 설계 문서 작성

AI가 대상 도메인을 분석하고, 사용자와 대화하며 DCAD(DynamoDB Conversion Architecture Document) 템플릿을 섹션별로 채워 완성한다.

## Usage

```
/workflow:dcad-design order                              # order 도메인 DCAD 신규 작성
/workflow:dcad-design payment ./dynamodb-design/payment_dcad.md   # 기존 문서 이어서 작성
/workflow:dcad-design product /path/to/product-service    # 특정 프로젝트 경로 지정
```

## Input

사용자 입력: `$ARGUMENTS`

- 도메인 이름 또는 프로젝트 경로 (필수)
- 기존 DCAD 문서 경로 (선택): 이미 작성 중인 문서가 있으면 이어서 진행

---

## 핵심 원칙

### 1. AI 주도 분석 + 사용자 확인

AI가 코드베이스, 엔티티, API, Repository를 먼저 분석하여 각 섹션의 초안을 작성한다. 사용자는 결과를 확인하고 피드백만 주면 된다.

### 2. 모호하면 반드시 질문

분석 중 다음 상황이 발생하면 추론으로 넘어가지 말고 사용자에게 질문한다:

| 상황 | 질문 예시 |
|------|----------|
| 엔티티 간 관계가 코드에서 명확하지 않을 때 | "Order와 Shipment의 관계가 1:1인가요, 1:N인가요?" |
| 비즈니스 규칙이 코드만으로 파악 불가할 때 | "환불 시 Payment 레코드를 새로 생성하나요, 기존 레코드를 수정하나요?" |
| 액세스 패턴의 호출 빈도를 추정할 수 없을 때 | "이 관리자 목록 조회는 얼마나 자주 호출되나요?" |
| GSI 설계에 대안이 2개 이상 존재할 때 | "A 방식과 B 방식이 있는데, 트레이드오프는 ~입니다. 어느 쪽을 선호하시나요?" |
| 전환 범위가 불명확할 때 | "이 도메인의 모든 테이블을 전환하나요, 특정 테이블만 대상인가요?" |

### 3. 섹션별 점진적 진행

한 번에 전체 문서를 생성하지 않는다. 섹션별로 작성 → 사용자 확인 → 다음 섹션 순서로 진행한다. 이전 섹션의 피드백이 이후 섹션에 반영되어야 하기 때문이다.

## 8. 체크리스트

- [ ] 모든 액세스 패턴이 Table 또는 GSI로 커버되는가

- [ ] 아이템 크기가 400KB를 초과하지 않는가

## 워크플로우

### Phase 0: 초기화

1. `$ARGUMENTS`에서 도메인명/프로젝트 경로를 파싱한다
2. 기존 DCAD 문서 경로가 있으면 읽어서 현재 진행 상태를 파악한다
3. 기존 문서가 없으면 위의 템플릿을 기반으로 빈 문서를 생성한다
4. 산출물 저장 경로를 결정한다:
   - 기존 문서가 있으면 해당 경로에 덮어쓰기
   - 없으면 `{프로젝트루트}/dynamodb-design/{도메인명}_dcad.md`에 생성
   - 디렉토리가 없으면 생성

### Phase 1: 도메인 개요 (섹션 1)

**AI 분석**:
- 프로젝트의 `CLAUDE.md`, `README.md`, `docs/` 등에서 도메인 설명 추출
- `build.gradle.kts` 또는 `pom.xml`에서 DB 드라이버 확인 → 현재 데이터 저장소 추정
- 엔티티 수, Repository 수로 데이터 규모 힌트 파악

**사용자에게 질문할 항목**:
- 전환 사유 (AI가 추정하기 어려움)
- 예상 데이터 규모 (비즈니스 수치)
- 읽기/쓰기 비율
- 지연시간 요구사항

**산출물**: 섹션 1 작성 → 사용자 확인

### Phase 2: 엔티티 관계 (섹션 2)

**AI 분석**:
- JPA `@OneToMany`, `@ManyToOne`, `@OneToOne` 등 관계 어노테이션에서 ERD 역산
- 테이블 간 카디널리티(1:1, 1:N, N:M) 파악
- Aggregate 경계 식별

> 필드 상세는 기술하지 않는다. 테이블 간 관계와 카디널리티만 간결하게 정리한다.

**사용자에게 질문할 항목**:
- 코드에 없는 암묵적 관계
- 엔티티 중 DDB 전환 대상에서 제외할 것

**산출물**: 섹션 2 작성 → 사용자 확인

### Phase 3: 유스케이스 + 액세스 패턴 (섹션 3, 4)

**AI 분석**:
- Controller/API 계층에서 엔드포인트 추출 → 유스케이스 도출
- Repository 메서드 시그니처에서 액세스 패턴 역산:
  - `findBy...` → Read 패턴
  - `save`, `deleteBy...` → Write 패턴
  - QueryDsl/JPQL/네이티브 쿼리 → WHERE, ORDER BY, JOIN 분석
- Service 계층에서 하나의 오퍼레이션이 여러 Repository를 호출하는 패턴 추적

**사용자에게 질문할 항목**:
- 각 유스케이스의 호출 빈도 (높음/중간/낮음)
- 관리자 전용 vs 고객 전용 구분
- 배치/스케줄 작업으로만 사용되는 패턴
- 코드에 존재하지만 실제로 사용되지 않는 API

> **`ddb-access-pattern` 스킬과의 연계**: 이미 `ddb-access-pattern` 스킬로 액세스 패턴을 추출한 문서(`docs/ddb-access-patterns/{도메인명}.md`)가 있으면 해당 문서를 읽어서 섹션 3, 4의 초안으로 활용한다. 중복 분석을 하지 않는다.

**산출물**: 섹션 3, 4 작성 → 사용자 확인

### Phase 4: 키 설계 (섹션 5)

**AI 분석**:
- 섹션 4의 액세스 패턴에서 PK/SK 후보 역산
- 패턴별로 Table/GSI 매핑 설계
- GSI 통합 가능 여부 검토 (복합 SK로 합칠 수 있는지)

**설계 원칙**:
- Access Pattern First: 엔티티 구조가 아닌 액세스 패턴에서 키를 역산한다
- 고카디널리티 PK 우선: 파티션 균등 분산을 위해 유일한 값(orderId 등)을 PK로 사용
- GSI 최소화: GSI 1개 추가 = 모든 쓰기에 복제 비용 발생
- nullable 경계: DDB PK/SK에 null 불가 → nullable 필드는 합성키 전략 필요

**GSI 백프레셔 검증** (GSI를 추가할 때마다 반드시 수행):

GSI에 쓰기가 복제될 때, GSI 파티션의 쓰기 용량이 부족하면 메인 테이블 쓰기까지 throttle된다. 에러가 GSI가 아닌 메인 테이블에서 발생하므로 원인 추적이 어렵다.

각 GSI에 대해 아래 3가지를 점검한다:

| 점검 항목 | 위험 신호 | 예시 |
|----------|----------|------|
| GSI PK 카디널리티 | 값의 종류가 수십 개 이하 | `STATUS#PENDING`, `BRAND#A` 등 — 특정 값에 쓰기 집중 |
| 쓰기 편중도 | GSI PK 특정 값에 전체 쓰기의 50% 이상 집중 | 메인 브랜드에 사용자 90% 몰림 |
| 스파이크 가능성 | 이벤트/배치로 순간 쓰기가 평소의 5배 이상 | 타임딜, 대량 가입 이벤트 |

위험 신호가 1개 이상이면 사용자에게 경고하고 대안을 번호로 제시한다:

1. **GSI PK 샤딩**: `BRAND_A#0` ~ `BRAND_A#9`처럼 접미사로 파티션 분산. 조회 시 N번 Query 후 클라이언트 병합 필요
2. **GSI 제거 + 메인 테이블 PK 활용**: PK prefix에 이미 해당 값이 포함되어 있으면 `begins_with`로 Query 가능. GSI 자체를 제거하여 백프레셔 원천 차단
3. **GSI 유지 + 허용된 리스크로 문서화**: 호출 빈도가 낮아 실질적 위험이 작은 경우 (예: 관리자 전용 조회), 체크리스트에 허용 사유를 명시

**대안이 있을 때**: 반드시 선택지를 번호로 매겨서 제시하고, 각각의 장단점과 트레이드오프를 설명한 뒤 사용자에게 선택을 받는다.

**산출물**: 섹션 5 작성 → 사용자 확인

### Phase 5: 용량 산정 (섹션 6)

**AI 주도 진행**:

1. 도메인의 RDB 테이블을 매핑하여 사용자에게 제시:
   > "아래 테이블들의 데이터 건수가 필요합니다."

2. 매핑 테이블과 SQL 쿼리를 생성하여 사용자에게 제공:
   - RDB 테이블 → DDB 아이템 타입 매핑표
   - 총 건수 + 최근 1개월 생성 건수 조회 쿼리 (여러 테이블 UNION ALL)

3. 사용자가 쿼리 결과를 입력하면 AI가 자동 산출:
   - 평균 아이템 크기 추정
   - 초당 쓰기/읽기 처리량 (기간별 건수에서 역산)
   - WCU/RCU 추정
   - 용량 모드 추천 (On-Demand vs Provisioned)
   - 월 예상 비용 개략 산출

**산출물**: 섹션 6 작성 → 사용자 확인

### Phase 6: 전환 시 고려사항 (섹션 7)

**AI 분석**:
- `@Transactional` 어노테이션이 있는 서비스 메서드 → 트랜잭션 요구사항 도출
- RDB JOIN으로 구현된 쿼리 중 DDB로 직접 전환이 어려운 항목 식별
- 집계 쿼리 (COUNT, SUM, GROUP BY) → DDB 전환 불가 후보

**사용자에게 질문할 항목**:
- 마이그레이션 방식 선호 (빅뱅 / 점진적 이중 쓰기 / CDC)
- 다운타임 허용 여부
- 롤백 시나리오

**산출물**: 섹션 7 작성 → 사용자 확인

### Phase 7: 체크리스트 + 최종 문서 (섹션 8)

1. 체크리스트 각 항목을 AI가 검증하여 통과/미통과 표시
2. 미통과 항목이 있으면 해당 섹션으로 돌아가서 보완
3. **DynamoDB MCP를 통한 구조 검증** (아래 절차 참조)
4. 전체 문서를 파일로 저장하고 경로를 사용자에게 안내

#### DynamoDB MCP 검증 절차

DCAD 문서의 키 설계(섹션 5)를 DynamoDB MCP 도구로 실제 검증한다. 검증은 3단계로 진행한다.

**Step 7-3a: 데이터 모델 JSON 생성**

섹션 5의 키 설계와 섹션 4의 액세스 패턴을 기반으로 `dynamodb_data_model.json`을 생성한다:
- 저장 경로: `{산출물 디렉토리}/dynamodb_data_model.json`
- 테이블 정의 (PK/SK, GSI), 테스트 아이템, 액세스 패턴의 AWS CLI 구현을 포함

> 포맷이 불확실하면 `mcp__awslabs_dynamodb-mcp-server__dynamodb_data_model_validation`을 workspace_dir만 지정하여 호출한다. JSON 파일이 없으면 생성 가이드를 반환해주므로 그 가이드를 따른다.

**Step 7-3b: 모델 검증 실행**

`mcp__awslabs_dynamodb-mcp-server__dynamodb_data_model_validation` 호출:
- `workspace_dir`: 산출물 디렉토리 절대 경로
- 이 도구가 DynamoDB Local을 자동 셋업하고, 테이블 생성 + 데이터 삽입 + 전체 액세스 패턴 실행을 수행한다
- 결과는 `dynamodb_model_validation.json`에 저장됨

**검증 실패 시**: 실패한 액세스 패턴을 사용자에게 보여주고, 섹션 5 키 설계를 수정한 뒤 재검증한다.

**Step 7-3c: 비용 산정 (Phase 5 용량 데이터가 있는 경우)**

`mcp__awslabs_dynamodb-mcp-server__compute_performances_and_costs` 호출:
- 섹션 4의 액세스 패턴 + 섹션 6의 데이터 규모를 입력으로 WCU/RCU/월 비용을 산출
- 결과를 섹션 6.2에 반영

> **Docker/Podman 미설치 시**: DynamoDB Local 셋업이 실패할 수 있다. 이 경우 사용자에게 안내하고 MCP 검증을 건너뛴다. 체크리스트의 "DynamoDB MCP를 통한 구조 검증 완료" 항목은 미통과로 표시하되, 나머지 항목은 계속 진행한다.

## 이어서 작성하기

기존 DCAD 문서가 주어진 경우:

1. 문서를 읽고 각 섹션의 완성도를 평가한다
2. 비어있거나 불완전한 섹션을 식별한다
3. 해당 섹션의 Phase부터 이어서 진행한다
4. 이미 완성된 섹션은 건너뛴다 — **단, 아래 의존성 매트릭스에 따라 영향받는 하위 섹션은 반드시 재검토한다**

### Phase 간 의존성 매트릭스

섹션을 수정하면 화살표 방향의 하위 섹션에 연쇄 영향이 발생한다. 재검토가 필요한 범위를 판단할 때 이 매트릭스를 참조한다.

```
섹션 2 (엔티티 관계)
  └→ 섹션 3 (유스케이스) → 섹션 4 (액세스 패턴)
                              └→ 섹션 5 (키 설계) → 섹션 6 (용량 산정)
                              └→ 섹션 7 (전환 고려사항)
```

| 변경된 섹션 | 재검토 필수 | 재검토 조건부 (변경 내용에 따라) |
|------------|-----------|-------------------------------|
| 섹션 2 (엔티티) | 3, 4 | 5, 6, 7 |
| 섹션 3 (유스케이스) | 4 | 5, 6 |
| 섹션 4 (액세스 패턴) | 5 | 6, 7 |
| 섹션 5 (키 설계) | 6 | 7 (트랜잭션 요구사항 변경 시) |
| 섹션 6 (용량 산정) | — | — (독립적) |
| 섹션 7 (전환 고려) | — | 5 (전환 불가 항목이 키 설계에 영향 줄 때) |

**재검토 절차**: 영향받는 섹션을 사용자에게 목록으로 보여주고, 각 섹션을 업데이트할지 확인한 뒤 진행한다. 사용자가 "전부 업데이트"라고 하면 의존 순서대로 순차 진행한다.

사용자에게 현재 상태를 요약하고, 다음에 진행할 섹션을 안내한다:
> "현재 섹션 1~3까지 완성되어 있습니다. 섹션 4(액세스 패턴 도출)부터 이어서 진행하겠습니다."

## Gotchas

실사용에서 발생할 수 있는 실패 패턴:

| Gotcha | 증상 | 대응 |
|--------|------|------|
| **JPA 어노테이션 미사용 프로젝트** | Phase 2에서 `@OneToMany` 등을 찾지 못함 | MyBatis, Exposed 등 다른 ORM 패턴을 탐색. SQL 매퍼 XML이나 테이블 DDL에서 관계 역산 |
| **ddb-access-pattern 문서 경로 불일치** | Phase 3에서 `docs/ddb-access-patterns/{도메인명}.md` 파일이 없음 | 경로가 존재하지 않으면 건너뛰고 코드베이스에서 직접 분석. 파일 존재 여부를 먼저 확인한 후 참조 |
| **DynamoDB Local 환경 미설치** | Phase 7 MCP 검증에서 Docker/Podman을 찾지 못함 | 사용자에게 안내하고 MCP 검증 단계를 건너뜀. 체크리스트에 미통과로 기록 |
| **대규모 도메인 (엔티티 20개+)** | Phase 2~4가 과도하게 길어짐 | Aggregate 단위로 서브그룹을 나누어 진행. 사용자에게 우선순위를 확인하여 핵심 엔티티부터 처리 |
| **멀티 모듈 프로젝트** | 도메인 코드가 여러 모듈에 분산되어 분석 범위가 불명확 | Phase 0에서 대상 모듈 경로를 사용자에게 확인. `build.gradle.kts`의 모듈 구조를 먼저 파악 |
| **기존 문서 포맷 불일치** | 이어서 작성하기에서 템플릿과 다른 형식의 기존 문서가 입력됨 | 섹션 매핑을 최선으로 시도하되, 매핑이 불확실한 부분은 사용자에게 질문 |
