# /worklog — 작업일지용 커밋 요약

오늘 진행한 작업을 작업일지에 적을 수 있도록 요약한다.

인자로 기간이나 추가 지시가 들어오면 반영한다 (`$ARGUMENTS`). 인자가 없으면 오늘 날짜 기준으로 동작한다.

## 수집 방법

1. `git log`로 오늘 날짜(또는 지정된 기간)의 커밋을 확인한다.
2. 각 커밋의 제목과 주요 변경 내용을 파악한다.
3. 연관된 커밋끼리 묶어서 하나의 작업 단위로 식별한다.

## 출력 형식

- 최상위 제목: 날짜 + 작업일지
- 중간 제목: 작업의 큰 주제 한 줄 (예: "기능명 — 무엇을 했는지")
- 내용: 각 작업을 **한 줄씩 불릿**으로 작성

## 작성 규칙

- 각 불릿은 한 문장으로 끝낸다. 하위 불릿이나 상세 설명을 붙이지 않는다.
- 커밋 해시, 파일 경로, 함수명, 변수명은 적지 않는다.
- 프로젝트 내부에서만 통용되는 약어·문서명·결정번호(예: ADR-006, T3-4, R1)는 적지 않는다.
- 기술 용어 중 외부인이 모를 법한 것(예: Outbox 패턴, B-2 구조, Relay)은 풀어서 설명하거나 생략한다.
- 추상적 표현 대신 "무엇을 했는지"가 동사로 드러나게 쓴다 (예: "조사", "작성", "정리", "등록").
- 완료된 사실만 적는다. 배경·동기·다음 단계는 적지 않는다 (사용자가 요청하면 별도 섹션으로 추가).
- 3~5줄 이내로 압축한다. 그 이상이면 작업 단위를 더 큰 묶음으로 합친다.

## 톤

- 프로젝트 컨텍스트를 모르는 사람이 읽어도 "무슨 일을 했는지" 이해되는 수준
- 기술 세부보다 작업의 목적과 결과 중심

## 예시 출력

```
# 2026-04-22 작업일지

## 결제 시스템 분리 — 환불 정책 통합

- 환불·결제실패 관련 코드 경로의 현재 구현 상태를 전수 조사해 문서화
- 외부 결제 호출이 동일한 구조를 따르도록 통합 정책 문서를 작성
- 프론트 주문 취소 버튼의 이중 클릭 방어가 부족함을 확인하고 후속 개선 과제로 등록
```
