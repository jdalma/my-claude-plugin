🔑왜 spool과 mailbox를 분리하나?

mailbox 파일은 소유자(수신자)만 쓴다는 불변식을 지키기 위해서다. 보내는 워커는 절대 받는 워커의 mailbox를 직접 건드리지 않고, 자기 몫의 spool 파일만 떨군다. 받는 워커가 그것을 자기 mailbox로 흡수한다. 그래서 여러 워커가 동시에 같은 수신자에게 보내도 (파일명이 고유 id라) 서로 덮어쓰지 않는다.

파일이 사는 곳

~/.my-team/sessions/demo/ ├── manifest.json # 팀 메타 (status/shutdown이 읽음) ├── events.jsonl # 팀 전체 메시지 타임라인 (monitor가 tail) ├── mailbox/ │ ├── A.json # A 소유: A의 inbox + A의 sent_pending │ └── B.json # B 소유: B의 inbox + B의 sent_pending ├── incoming-spool/ │ ├── A/<id>.json # A에게 온 메시지 (수신 대기, 흡수 후 삭제) │ └── B/<id>.json # B에게 온 메시지 (수신 대기, 흡수 후 삭제) └── archive/ ├── A.jsonl # A의 송수신 영구 기록 (out/in) └── B.jsonl # B의 송수신 영구 기록 (out/in)

설계 근거 — 자주 나오는 질문

Q.incoming-spool이 왜 필요한가? 그냥 B의 mailbox에 바로 쓰면 안 되나?

동시에 여러 워커가 한 수신자에게 보낼 때 메시지가 사라지는 것(lost update)을 막기 위해서다.

mailbox에 메시지를 추가하는 건 append가 아니라 read → modify → write 전체다. JSON 객체(map)는 부분 추가가 불가능해서, 키 하나를 넣으려면 파일 전체를 읽고 통째로 다시 써야 한다(atomicWriteJson = 전체 덮어쓰기). 그래서 A와 C가 동시에 B에게 보내면:

A: B.json 읽음 (inbox = {}) C: B.json 읽음 (inbox = {}) A: inbox에 msgA 추가 → {msgA} C: inbox에 msgC 추가 → {msgC} A: B.json 통째로 씀 → {msgA} C: B.json 통째로 씀 → {msgC} ← msgA 덮어씀! 최종: B.json = {msgC} ← msgA 영구 소실

atomicWriteJson은 "쓰는 도중 깨진 파일이 보이는 것(torn write)"은 막지만, 위 lost update는 못 막는다 — 각 쓰기는 완결됐고 더 늦은 쓰기가 앞 결과를 덮은 것뿐이다.

spool은 메시지 1건 = 파일 1개(파일명 = 고유 message_id)다. 송신자는 자기 id로 된 새 파일 하나만 만들 뿐 read-modify-write를 안 하므로 충돌이 원천 불가능하다. 덕분에 "mailbox는 소유자만 쓴다"는 불변식이 지켜진다 — 송신자는 spool에만 쓰고, mailbox는 수신자 한 명만 쓰는 단일 writer가 된다.

💡 비유 — JSON map은 "공책 한 페이지를 통째로 베껴 다시 쓰는 것". 두 사람이 같은 페이지를 동시에 베끼면 늦게 끝낸 쪽이 상대 글을 지운다. spool은 "각자 포스트잇 한 장씩 붙이기" — 서로 겹칠 일이 없다.

※ 현재 운영 모델(사람 1명이 명령을 직렬 실행)에선 동시 송신이 실제로 안 일어나 이 race가 아직 실현되진 않는다. 워커끼리 서로를 오케스트레이션하는 단계를 대비한 선제적 안전장치다.

Q.B는 어떤 게 자기에게 온 메시지인지 어떻게 구분하나?

구분하지 않는다. 디렉토리 경로가 이미 수신자별로 나뉘어 있다.

spool 경로는 incoming-spool/<to_worker>/<message_id>.json 구조다. 수신자 이름이 디렉토리명에 박혀 있다. A가 B에게 보낸 메시지는 처음부터 incoming-spool/B/ 안에만 들어가고, C가 D에게 보낸 건 incoming-spool/D/로 — 물리적으로 다른 폴더다.

B가 mailbox-list를 호출하면 worker:"B" 인자로 incoming-spool/B/만 readdir로 훑는다. 다른 워커의 폴더는 쳐다보지도 않는다. 그래서 B 입장에선 "이게 내 거 맞나?"를 판단할 필요가 없다 — 자기 폴더 안에 있으면 전부 자기 것이다.

분류는 받는 쪽이 아니라 보내는 쪽에서 끝난다. 송신 시점에 to_worker:"B" → incoming-spool/B/로 라우팅되기 때문이다.

💡 비유 — 이름표별로 나뉜 아파트 우편함. 집배원(A)이 "B호" 칸에 편지를 넣고 초인종(벨)을 누르면, B는 자기 칸만 열어보면 된다. "내 거 맞나" 확인할 필요 없이 B 칸에 든 건 다 B 것이다. 벨 문자열 new-message:A도 "어떤 메시지"가 아니라 "A가 뭔가 보냈으니 확인해"라는 신호일 뿐이다.
Q.spool 안의 메시지를 mailbox로 옮기는 건 누가 담당하나?

전적으로 수신 워커가 담당한다. 그리고 이게 동시성 안전의 핵심이다.

작업담당함수
spool에 파일 떨구기송신 워커 (A)dropSpoolMessage
spool → mailbox 흡수수신 워커 (B)absorbIncomingSpool

송신자는 incoming-spool/B/에 파일을 넣기만 한다. 그것을 mailbox로 옮기는 건 B가 mailbox-list를 호출할 때 일어난다:

absorbIncomingSpool(team, B, ...):
  1. incoming-spool/B/ 의 모든 *.json 을 readdir
  2. mailbox/B.json 을 읽어서
  3. 각 spool 파일 내용을 inbox 맵에 추가
  4. mailbox/B.json 을 통째로 다시 씀   // ★ B만 하는 쓰기
  5. 흡수한 spool 파일은 unlink (삭제)

4번(mailbox 쓰기)을 오직 B만 수행하기 때문에 "mailbox는 소유자만 쓴다"는 불변식이 지켜진다. 흡수를 수신자에게 맡긴 것 자체가 설계다 — 송신자들은 충돌 없는 spool 파일만 만들고, 그걸 mailbox로 합치는 단 하나의 직렬 지점을 수신자 한 명에게 몰아줘서 동시 쓰기 충돌을 원천 차단한다.

incoming-spool/B/의 .json 파일은 A가 보낸 것·C가 보낸 것 구분 없이 전부 한 번에 흡수되고, 흡수 후 각 파일은 unlink되어 다음 폴링 때 중복 흡수되지 않는다.

※ 벨 자체는 흡수를 하지 않는다. 벨은 "확인해" 신호일 뿐이고 실제 흡수는 B가 mailbox-list를 호출해야 일어난다. 그래서 벨을 놓쳐도 다음 사이클 폴링이 빠짐없이 흡수한다.

"요청"이지만 강제 실행이 아니다

⏳A는 B를 실행시키지 않는다 — 수신함에 넣을 뿐

A가 보낸다고 B가 즉시 멈추고 처리하는 게 아니다. A는 메시지 파일을 spool에 떨구고 자기 일을 계속한다. B가 그것을 언제 흡수·처리할지는 B의 작업 사이클에 달려 있다. 시스템에는 deadline도 timeout도 없다.

그래서 받는 워커의 self-poll(자가 폴링)이 진짜 전달 보장이다. pane 벨은 "빨리 보라"는 힌트일 뿐, 놓쳐도 다음 사이클의 mailbox-list가 spool을 흡수하므로 메시지 자체는 잃지 않는다. (잃는 건 즉시성뿐)
